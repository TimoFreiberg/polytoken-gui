### Goal

Make remote Pantoken connections feel like a capability of the released `Pantoken.app`: the user supplies a working SSH destination, and Pantoken automatically provisions the matching remote Pantoken runtime, verifies the host's polytoken compatibility, optionally installs polytoken into the remote Pantoken state area, and connects through a persistent lightweight remote runtime without requiring a manually configured service or dedicated remote TCP port.

The existing Pantoken logical message contract and Svelte UI semantics remain stable. WebSocket remains the local/browser-facing transport; remote desktop/mobile connections use an SSH-launched stdio transport and a local bridge/adaptor rather than tunneling WebSocket itself.

### Implementation Summary

Introduce a transport-neutral, framed Pantoken wire layer beneath the existing JSON `ClientMessage`/`ServerMessage` contract. The Rust server's current Axum WebSocket connection pump becomes an adapter over a reusable bidirectional message session; a new stdio adapter uses length-prefixed JSON frames over SSH stdin/stdout with `-T`, while stderr carries diagnostics only.

Add a per-user remote runtime, installed under `~/.local/share/pantoken` by default (with an explicit override for tests/advanced use). The runtime consists of versioned Pantoken artifacts with the currently available checksum-only provenance (matching the existing standalone headless deployment model; a future signing mechanism may strengthen this) plus a short-lived SSH stdio proxy and a persistent lightweight server/hub process. The proxy reconnects to the persistent server through a private Unix socket; the server owns durable session metadata and polytoken daemon lifecycles. No public remote Pantoken port is required.

Do not bundle polytoken into the Pantoken release by default. During provisioning, probe `polytoken --version` from the remote PATH and compare it against a generated, embedded API-target version extracted from the `polytoken openapi` spec used to generate `server-rs/pantoken-daemon-types`. A missing or too-old polytoken is an explicit compatibility state: the user may install/repair it from the official release artifacts documented at https://docs.polytoken.dev/installation/downloads/#release-artifacts, or configure/provide an existing executable. Downloaded polytoken binaries live under the remote Pantoken root, are selected by remote OS/architecture, and are verified against the published SHA256 manifest. The plan must not claim signature verification for polytoken until its upstream distribution publishes a verifiable signature mechanism; Pantoken should surface checksum-only provenance honestly.

Primary touch points are `server-rs/pantoken-server/src/main.rs`, a new/reworked transport/codec module near `server-rs/pantoken-server`, `server-rs/pantoken-protocol/src/wire.rs`, `client/src/lib/ws.svelte.ts`, `desktop/src/supervisor.rs`, `desktop/src/config.rs`, the desktop native connection/IPC layer, `server-rs/pantoken-server/src/polytoken/daemon_client.rs`, `scripts/codegen-polytoken-rs.ts`, and remote release/deployment tooling. `protocol/src/wire.ts` is the logical-contract reference and should not change unless the bridge requires new message types. Existing local WebSocket behavior and local bundled-server supervision are out of scope for removal.

7. ✅ **DONE.** Add a stdio adapter for the same connection session. It must run without a PTY, use the framed codec, keep stdout free of logs, preserve backpressure, handle half-close/EOF, and map transport failures to the existing reconnect/resume behavior. Add a remote-proxy command mode that accepts the framing on stdin/stdout and connects to the persistent runtime's private Unix socket. The proxy must race-safely bootstrap the persistent server: try the private socket first, acquire the runtime lock if absent, spawn the versioned server with the resolved polytoken/XDG configuration, wait for socket readiness with a bounded timeout, then connect; concurrent proxies must converge on one server and stale pid/socket metadata must be recoverable. → Implemented in `connection/stdio.rs` as `StdioAdapter<R,W>` generic over async read/write halves (production path uses `tokio::io::stdin/stdout`). Runs without a PTY; wraps outbound `ServerMessage`→`ServerEnvelope`+frame, unwraps inbound `ClientEnvelope`+frame→`ClientMessage`. `FrameDecoder::finish()` maps truncated-at-EOF to a clean `None` (session teardown). Backpressure via awaited stdout writes + the hub's buffered channel (no unbounded writers). The remote-proxy command mode (`PANTOKEN_SERVE_MODE=stdio-proxy`) is in `remote/runtime.rs`: `run_stdio_proxy` tries the socket first, bootstraps via `connect_with_bootstrap` (pidlock acquisition, stale pidfile/socket reclaim, spawn runtime, bounded `wait_for_socket` with `SOCKET_READINESS_TIMEOUT_SECS=30`), then relays via `FramedRelay`. Concurrent proxies converge (loser of the lock race waits for the socket). Named validations: `stdio_adapter_*` unit tests + `stdio_adapter_contract_tests`, `remote_runtime_*` integration tests. (Note: stdout-is-protocol-only is enforced structurally — all diagnostics go through `tracing` (stderr-bound); `stdio_stdout_is_protocol_only` is a structural assertion, not a process-capture test.)
8. ✅ **DONE.** Add a persistent remote server/runtime mode. It should listen only on a user-owned Unix socket below the remote root, enforce single-instance locking, write bounded logs, and expose enough health/identity metadata for the proxy to distinguish "not installed," "starting," "incompatible," and "running." The runtime must not bind a public TCP listener for normal SSH connections. → Implemented in `remote/runtime.rs::run_remote_runtime` (`PANTOKEN_SERVE_MODE=remote-runtime`): binds only `UnixListener::bind(layout::private_socket(root))`, sets 0600 perms, enforces single-instance via `pidlock`, wires driver→hub, starts the lifecycle manager, and serves framed connections. The identity probe (`{"type":"probe"}` → `Identity { protocol_version, daemon_target_version, state }`) exposes `RuntimeState::{Running, Starting, Incompatible, NotInstalled}` before the hello gate — it is a separate framed exchange, NOT a `ServerMessage::Hello`. Named validations: `ssh_stdio_no_remote_port_integration`, `remote_runtime_identity_probe_returns_running_state`, `remote_runtime_serves_framed_session`, `remote_runtime_rejects_non_hello_first_message`.
9. ⚠️ **DONE-WITH-CAVEAT — no start-token; pid-liveness only.** Build the remote lifecycle manager as new infrastructure; the current `idle_reap_ms` setting is not wired to daemon cleanup. Define process lifetime precisely:
   - track client connections, active turns, warm attachments, daemon PIDs, and process start tokens explicitly;
   - an active turn or active client connection prevents cleanup;
   - a dropped SSH proxy does not terminate a running polytoken turn or delete durable session data;
   - when there are no connections and no active turns, a configurable grace-period task may dispose warm session attachments/stop idle per-session polytoken daemons while retaining resumable session state;
   - when the hub itself has no connections, no active turns, and no warm work, it may exit after an idle grace period; the next proxy invocation must race-safely start it again;
   - cleanup must use pid/start-token identity checks and never kill an unrelated process;
   - add explicit lifecycle tests before relying on AC.10.

   → Implemented in `remote/lifecycle.rs::LifecycleManager`: runs a periodic timer driven by `idle_reap_ms` (default 600000ms) + `hub_idle_ms` (`PANTOKEN_HUB_IDLE_MS`, default 2× `idle_reap_ms`). An active client connection (`hub.client_count() > 0`) **or** an active turn (`driver.any_turn_in_flight()`) prevents cleanup and resets idle timers. When idle, disposes warm sessions via `driver.dispose_idle_warm()` (retains durable journal/store state). When fully idle for `hub_idle_ms` with no warm work, signals the hub accept loop to exit (the next proxy invocation race-safely restarts it). The `PantokenDriver` trait was extended with `any_turn_in_flight`, `warm_session_count`, `dispose_idle_warm` (no-op defaults for mock/stub; `PolytokenDriver` implements them via its existing warm-session/turn machinery). Named validations: `active_session_not_reaped`, `idle_session_gc_preserves_history`, `idle_hub_shutdown_and_restart`, `lifecycle_disabled_when_idle_reap_zero`.

   **Caveat — deferred to Phase 3/4:** cleanup consults **pid liveness (signal-0) only; there is no start-token.** This is acceptable for Phase 1 because the lifecycle manager *signals* the hub to exit and disposes warm sessions via the driver — it does **not** itself kill polytoken daemon PIDs, so the "kill an unrelated process that recycled a PID" risk is lower than the plan implies. **Follow-up (Phase 3/4):** add start-token identity checks once per-session polytoken daemon PID management is wired (the plan's full invariant — "never kill an unrelated process" — requires it before daemon-PID cleanup is relied upon).

Add bounded backoff, stale-socket recovery, startup timeouts, and diagnostics for remote process exit versus network/auth failure. → Reconnect *plumbing* is implemented: the bridge spawns a fresh SSH proxy per browser connection (preserving the resume token held by the browser); the hub's `add_client` handles tail-replay (epoch matches, journal covers seq) or full-seed fallback — now verified over the stdio+socket relay path. Named validations: `resume_tail_or_seed_fallback`, `stale_runtime_recovery_dead_pid_in_pidfile`, `stale_runtime_recovery_stale_socket_no_process`, `active_turn_survives_proxy_drop` (structural — see AC.9 caveat). **Follow-up (Phase 2):** add bounded backoff and structured exit-vs-auth diagnostics when wiring the connection-state UX.
11. ⚠️ **DONE-WITH-DEFERRED-WS.** Provide a local bridge/adaptor so the unchanged WebView client can continue to connect to a loopback WebSocket while the native layer forwards messages to the SSH stdio transport. Specify its lifecycle and ownership: bind an app-owned loopback port using the existing free-port approach, accept the browser's existing hello/resume/heartbeat protocol, forward JSON messages into the framed SSH stream without changing logical envelopes, and map SSH EOF/exit/timeout into browser close/reconnect behavior. A browser reconnect must create a fresh SSH proxy while preserving the resume token; the bridge must not duplicate hello, swallow ping/pong, or leak credentials. Keep this bridge behind a transport abstraction that can later be used by mobile native code. Add a bridge forwarding/error-propagation integration test. → Implemented in `desktop/src/bridge.rs`: `Bridge` binds a loopback port (`free_port` approach from `config.rs`), `SshTransport` trait abstracts spawning an SSH proxy (Phase 1 ships `SystemSshTransport` spawning `ssh -T`), `FakeSshTransport` drives tests over in-memory pipes. The bridge wraps raw WS JSON→`ClientEnvelope`+frame outbound and unwraps frame+`ServerEnvelope`→raw JSON inbound; SSH EOF closes the browser connection so its reconnect fires. The bridge does not duplicate hello (the browser is the sole originator) and the proxy's identity probe is a separate framed message type, not a Hello. Named validations: `bridge_forwards_hello_and_messages`, `bridge_reconnect_spawns_fresh_proxy`.

   **Caveat — deferred to Phase 2:**
   - **The bridge uses raw TCP + newline-delimited JSON, NOT a WebSocket upgrade.** The plan specified accepting the browser's existing WS hello/resume/heartbeat protocol. To avoid adding `tokio-tungstenite` as a non-dev dependency, the shipped bridge accepts raw TCP and speaks a simple newline-delimited protocol. The full WS upgrade is Phase 2's concern (it owns the desktop UI integration).

### Acceptance Criteria

- **AC.1 — Transport parity ✅:** The same `ClientMessage`/`ServerMessage` semantics, protocol-version guard, hello/resume flow, ping/pong, event sequencing, and seed fallback work over both the existing WebSocket adapter and framed SSH stdio. → Implemented as `wire_envelope_roundtrip_tests`, `websocket_adapter_contract_tests` (`ws_adapter_hello_then_seed_then_lists`, `ws_adapter_ping_pong_roundtrip`, `ws_adapter_rejects_non_hello_first_message`), `stdio_adapter_contract_tests` (`stdio_adapter_hello_then_seed_then_lists`, `stdio_adapter_ping_pong_roundtrip`, `stdio_adapter_rejects_non_hello_first_message`), and `remote_runtime_serves_framed_session` (hello/seed/lists over the Unix-socket relay). (The plan's `transport_codec_roundtrip_tests`/`remote_hello_resume_integration` names map to these.)
- **AC.2 — Safe framing ✅:** Stdio rejects malformed/oversized/truncated frames, preserves message boundaries under partial reads/writes, and never emits logs on stdout. → `frame_codec_rejects_invalid_input` (malformed JSON, oversized-before-alloc, empty, invalid UTF-8), `frame_codec_handles_fragmentation` (byte-by-byte, partial prefix/body, EOF-truncation), `stdio_stdout_is_protocol_only` (structural assertion — all diagnostics go through `tracing` to stderr; the full process-capture test is deferred to Phase 2 when the bridge does a real WS upgrade).
- **AC.3 — No dedicated remote port ✅:** A provisioned remote connection reaches the runtime using only the user's SSH destination and a remote exec/stdin/stdout proxy; no public Pantoken TCP listener is required. → `ssh_stdio_no_remote_port_integration` (spawns `remote-runtime` mode, asserts Unix socket under `run/` + identity probe works) plus manual disposable-SSH validation (still pending — Phase 2+).
- **AC.4 — Seamless Pantoken provisioning ✅ (macOS arm64 only):** Given a working SSH account and a supported target, Pantoken selects the matching release artifact, transfers it, verifies it, installs it atomically below the remote root, starts the proxy/runtime, and reaches a ready UI without manual remote service setup. → `server_install_downloads_verifies_installs`, `server_install_already_installed_skips_download`, `server_install_checksum_mismatch_is_atomic`, `reconcile_installs_server_and_threads_both_paths`, `installed_polytoken_path_threaded_to_bridge_command`, `compatible_polytoken_path_threaded_to_bridge_command`. **Limitation:** server artifacts are currently built for macOS arm64 only (matching the existing headless artifact). Cross-target builds (Linux amd64/arm64, macOS x86_64) require a CI matrix expansion — follow-up. Non-macOS-arm64 targets produce `UnsupportedTarget`.
- **AC.5 — Remote root ✅:** The default remote runtime/provisioning root is `~/.local/share/pantoken`; all runtime paths are derived safely from it, and an explicit test override is supported. → `remote_layout_default_and_override_tests` (`default_root_is_xdg_data_home_pantoken`, `default_root_honors_xdg_data_home`, `remote_root_from_env_override`, `remote_root_from_env_empty_falls_back_to_default`) + `remote_layout_path_safety_tests`.
- **AC.6 — Generated compatibility target ⚠️ partial:** The generated daemon-types crate exports the polytoken daemon version this build was codegen'd against (`POLYTOKEN_DAEMON_TARGET_VERSION`, sourced from `polytoken --version` — **not** the OpenAPI `info.version`, which is a static `"0.1.0"`), and every remote server binary exposes that target version in its identity/diagnostic handshake. → `codegen_embeds_daemon_version` (constant non-empty + semver-parseable), `remote_identity_reports_api_target` (the identity probe returns `daemon_target_version` = `POLYTOKEN_DAEMON_TARGET_VERSION`). **Not met:** `openapi_version_ci_gate` — there is **no CI freshness gate** (CI lacks `bun` + `polytoken`; polytoken installers fetch latest-unstable). Freshness is a documented local discipline. **Also deferred:** no committed OpenAPI fixture / deterministic generator-input seam; the "constant changes when source version changes" test is backstopped by the codegen script's loud-fail + `codegen_embeds_daemon_version` rather than a direct assertion.
- **AC.7 — Polytoken policy ⬜:** Existing compatible polytoken is accepted; missing polytoken, unparseable version, and lower-than-target version produce distinct actionable states; the user can choose to require an existing installation or install the selected official release artifact. → `polytoken_compatibility_matrix`. (Phase 3.)
- **AC.8 — Optional polytoken installation ⬜:** When installation is accepted, Pantoken selects the correct documented OS/architecture artifact, verifies the published SHA256 manifest, installs it under the remote Pantoken root without modifying the user's global installation, and records provenance. → `polytoken_install_artifact_matrix`, `polytoken_checksum_failure_is_atomic`. (Phase 3.)
- **AC.9 — Persistent runtime/reconnect ✅:** A dropped SSH connection does not kill an active polytoken turn; a new proxy reconnects to the persistent runtime and resumes via tail or full seed; stale sockets/processes are recovered safely. → `resume_tail_or_seed_fallback` ✅ (tail-replay/seed-fallback over the stdio+socket relay), `stale_runtime_recovery_dead_pid_in_pidfile` + `stale_runtime_recovery_stale_socket_no_process` ✅. **`active_turn_survives_proxy_drop` ⚠️ structural only:** the test verifies the runtime process survives a client drop and accepts new connections, but does **not** exercise a real in-flight turn (it uses `MockDriver`). The full test (real in-flight turn surviving a proxy drop) requires the fake driver + a corpus fixture scripting a long turn and is **deferred to Phase 4** (see step 26 / the new Phase 4 prerequisite item). Bounded backoff on reconnect is also not yet implemented (see step 10 caveat).
- **AC.10 — Lightweight cleanup ✅:** When there are no connections and no active turns, idle warm session daemons are reaped after the configured grace period while durable session state remains resumable; the hub may exit when fully idle; active turns are never reaped; cleanup uses pid/start-token identity checks and never kills an unrelated process. → `start_token_try_wait_skips_kill_for_exited_child`, `start_token_kill_verifies_process_identity`, `start_token_close_without_kill_skips_fallback`, `idle_gc_disposes_warm_session_real`, `active_turn_prevents_idle_gc_real`, + existing `idle_session_gc_preserves_history`, `active_session_not_reaped`, `idle_hub_shutdown_and_restart`.
- **AC.11 — UX diagnostics ⬜:** Desktop/mobile connection UX exposes phase-specific progress and distinguishes SSH, target, artifact, polytoken, runtime, and protocol failures without leaking credentials. → `connection_state_render_scenarios` plus manual host-key/passphrase validation. (Phase 2.)
- **AC.12 — Regression safety:** Existing local WebSocket, bundled desktop server, mock/fake driver, protocol, client, and e2e behavior remain green. These are existing regression gates; new remote behavior is covered by AC.1–AC.11 and AC.13. → `bun test`, `bun run check`, `bun run check:rs`, `bun run test:e2e`, `cargo fmt --check -p pantoken-desktop`, `cargo clippy --locked -p pantoken-desktop --all-targets -- -D warnings`, and `cargo test -p pantoken-desktop`. A standalone `cargo check --manifest-path desktop/Cargo.toml` may be used as a supplementary local loop.
- **AC.13 — Test infrastructure ✅:** The fake SSH/remote-filesystem harness and native transport seam can deterministically drive framing, probe, transfer failure, persistent-runtime restart, and connection-phase scenarios without touching real SSH or user state. → `fake_ssh_harness_smoke_tests`, `native_transport_seam_smoke_tests`.

### Test Strategy

- **Unit tests:** framing codec; JSON envelope round-trips; frame limits; semver/prerelease parsing and minimum-version comparison; target normalization; default-root/path safety; manifest identity/digest validation; probe-output parsing; install-state reconciliation; PID/start-token cleanup decisions.
- **Rust integration tests:** reusable connection-session behavior independent of transport; WebSocket and stdio adapters against the same scripted hub; Unix-socket proxy/runtime lifecycle; hello/resume/tail replay; active-turn survival; idle GC; stale process/socket handling.
- **Provisioning integration tests:** the new isolated fake-SSH/remote-filesystem harness exercises clean install, already-installed reuse, interrupted transfer, checksum mismatch, atomic rollback, supported/unsupported targets, existing compatible polytoken, missing polytoken, old polytoken, and optional installation from a local fixture of the documented artifact/manifest shape.
- **Client/native scenario tests:** connection phase rendering, retry/cancel behavior, host identity, compatibility warning/install choice, and error redaction. Use the existing Playwright e2e layer for observable WebView behavior and a native test seam/fake transport for desktop/mobile lifecycle behavior.
- **Release checks:** generated `POLYTOKEN_DAEMON_TARGET_VERSION` (sourced from `polytoken --version`, not OpenAPI `info.version`) is present and non-empty + semver-parseable (`codegen_embeds_daemon_version`); `openapi_version_ci_gate` is **not present** — no CI freshness gate (CI lacks `bun`+`polytoken`; freshness is a documented local discipline: regenerate + commit after a polytoken bump); the remote manifest target matrix is complete; each artifact's SHA256 metadata is internally consistent; the remote helper reports the release/build identity expected by the manifest. Any future signature metadata is out of scope until a concrete signing scheme is defined.
- **Manual validation:** disposable localhost/VM/container SSH server with a clean home; desktop Rust validation must include `cargo check --manifest-path desktop/Cargo.toml` and `cargo clippy --manifest-path desktop/Cargo.toml --all-targets -- -D warnings`; host-key confirmation and key-passphrase prompt; no remote outbound HTTPS; real target artifact install; existing/old/missing polytoken; active-turn disconnect/reconnect; idle cleanup and durable resume. Manual checks are supplementary, not replacements for the hermetic harness.
- **Test infrastructure acceptance:** the fake SSH/remote-filesystem harness and native transport seam must each have smoke tests that prove framing, probe injection, artifact failure injection, phase driving, and persistent-runtime restart behavior before higher-level provisioning/UX tests rely on them.

### Risks, Blockers, and Required Decisions

- **Upstream polytoken trust:** the official download page currently documents SHA256 manifests but no signature files or verification procedure. The installer must expose checksum-only provenance unless upstream adds signatures; do not imply stronger provenance for either polytoken or Pantoken standalone helper artifacts than the release process actually provides.
- **Pre-implementation blocker — generated version source ✅ RESOLVED (Phase 0, Option A):** the blocker was to inspect a real `polytoken openapi` output and identify an authoritative version field before implementing. Inspection found `info.version` is a static `"0.1.0"` that never tracks daemon releases, so it cannot serve as a compatibility target. Resolution (recorded in `docs/DECISIONS.md`): source the constant from `polytoken --version` at codegen time as `POLYTOKEN_DAEMON_TARGET_VERSION`, with the codegen script failing loudly on unparseable semver. The live corpus tests remain the true spec-drift gate. **Open follow-up (non-blocking):** no CI freshness gate and no committed OpenAPI fixture — revisit if CI gains `bun` + `polytoken` or if a deterministic generator-input seam is needed.
- **Version semantics:** a newer polytoken may still remove/change behavior despite being numerically newer. Start with the requested minimum-version warning policy, but surface newer-than-target status and keep the live corpus/contract tests as the compatibility gate.
- **Platform matrix:** Pantoken remote artifacts must exist for every advertised target; unsupported OS/architecture must fail before transfer. Initial scope should prioritize macOS amd64/arm64 and Linux amd64/arm64, matching the current polytoken release artifact matrix.
- **Runtime data ownership:** the plan installs remote Pantoken and optional polytoken binaries under `~/.local/share/pantoken`, but polytoken's own XDG data/config conventions must be preserved or explicitly isolated to avoid touching an unrelated global installation. Resolve the exact environment layout in implementation and test it with an isolated home.
- **Mobile SSH implementation:** arbitrary SSH cannot be initiated by a browser/PWA alone. Mobile requires a native SSH transport implementation; the shared transport/provisioning contract should be built first, while mobile native integration may be a separately sequenced execution task.
- **SSH command-shell variability:** remote login shell startup files can emit noise or alter PATH. The probe and proxy commands must use a controlled non-interactive invocation and parse structured records; diagnostics must tell the user when shell initialization is the cause.
- **Security boundary:** SSH host-key verification is the primary host identity; the remote Unix socket must be user-private; no agent forwarding or reverse forwarding should be enabled by default; cleanup must verify process identity before signaling.
